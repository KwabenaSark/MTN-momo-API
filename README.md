## API Reference

### Auth
```
POST /auth/login          → { token, user }
GET  /auth/me             → { user }          (requires token)
```

### Payments
```
POST /payments            → 202 PENDING        (requires Idempotency-Key header)
GET  /payments/:id        → transaction object
POST /payments/:id/refund → 202 PENDING        (requires Idempotency-Key header)
GET  /payments            → paginated list     (?status=SUCCESSFUL&limit=20&offset=0)
```

### Webhooks
```
POST /webhooks/register   → { signingSecret }
GET  /webhooks/config     → current config (secret redacted)
GET  /webhooks/deliveries → delivery log
```

### Admin (admin role only)
```
GET /admin/transactions   → all transactions (?status=&merchantId=)
GET /admin/metrics        → approval rate, volume, webhook stats
GET /admin/merchants      → merchant list
```

## Architecture Decisions

### 1. Idempotent Payments via `Idempotency-Key`

**The problem:** Mobile networks in West Africa have high latency and frequent timeouts. Without idempotency, a client that times out and retries will debit the subscriber twice.

**The solution:** Every `POST /payments` request requires an `Idempotency-Key` header (a UUID generated by the client). The server:

1. On first request: stores the key as `IN_FLIGHT`, processes normally, caches the response.
2. On retry with same key: returns the cached response with `X-Idempotent-Replay: true` — no re-processing.
3. On concurrent duplicate: returns `409 Conflict` — the original is still processing.

**Why this matters:** This is exactly the mechanism MTN MoMo's API enforces via `X-Reference-Id`. Idempotency keys are scoped per merchant so `MERCH-001::abc` never collides with `MERCH-002::abc`.

**Production note:** In production, idempotency keys would be stored in Redis (with a 24-hour TTL), not in-memory. Redis allows atomic `SET NX` to safely handle concurrent requests from multiple API server instances.

```bash
# First request — processes normally
curl -X POST http://localhost:3000/payments \
  -H "Authorization: Bearer $TOKEN" \
  -H "Idempotency-Key: a3f4c2d1-..." \
  -H "Content-Type: application/json" \
  -d '{"amount": 50, "phoneNumber": "0241234567"}'

# Identical retry — returns same transactionId, no double charge
curl -X POST http://localhost:3000/payments \
  -H "Idempotency-Key: a3f4c2d1-..."  ← same key
  ...
# Response includes: "X-Idempotent-Replay: true"
```

### 2. Async Two-Phase Payment Flow

**The problem:** MoMo payments are not synchronous. The subscriber must respond to a USSD push prompt on their handset, which can take seconds or minutes.

**The solution:** Payment initiation returns `202 Accepted` with `status: PENDING`. The final result (SUCCESSFUL / FAILED) is delivered via webhook callback. This mirrors the real MTN MoMo API pattern exactly.

```
Client → POST /payments → 202 PENDING
                            ↓
                  [Subscriber approves/rejects USSD prompt]
                            ↓
Provider → POST merchant-webhook-url → { status: "SUCCESSFUL" }
```

The simulation fires a `setTimeout` with a configurable delay (`PROVIDER_SIM_DELAY_MS`) and an 85% approval rate to produce realistic outcomes including failures.

---

### 3. RBAC via Middleware Factory

**Design:** A single `rbac(...roles)` factory returns a middleware function. This keeps route definitions readable and eliminates repetition.

```javascript
// Route definition reads like a permission spec
router.get('/admin/metrics', authenticate, rbac('admin'), handler)
router.post('/payments',     authenticate, rbac('merchant', 'admin'), handler)
```

**Merchant isolation:** Merchants can only read their own transactions. If merchant MERCH-001 requests `GET /payments/:id` for a transaction owned by MERCH-002, they receive `403 Forbidden` — not `404`. Returning 403 (not 404) is intentional: 404 would let merchants enumerate other merchants' transaction IDs.

---

### 4. Webhook Delivery with Retry

**Pattern:** Mirrors the retry schedule used by Stripe and Paystack.

```
Attempt 1: immediate
Attempt 2: +5 seconds
Attempt 3: +30 seconds
Attempt 4: +5 minutes
→ EXHAUSTED (dead-letter)
```

**Security:** Payloads are signed with `HMAC-SHA256` using a per-merchant secret. Merchants verify the `X-Signature` header before processing callbacks — this prevents spoofed webhooks from triggering fulfilment.

```javascript
// Merchant verification (example)
const expected = 'sha256=' + crypto.createHmac('sha256', signingSecret)
  .update(rawBody).digest('hex');
if (expected !== req.headers['x-signature']) reject();
```

**Production note:** In production, retries would be handled by a message queue (SQS, BullMQ) rather than `setTimeout` — this survives server restarts and scales across multiple workers.

---

### 5. Provider Detection from Phone Number

Ghana network prefixes are used to route payments to the correct provider:

| Prefix       | Provider          |
|--------------|-------------------|
| 024, 054, 055, 059 | MTN MoMo    |
| 020, 050     | Vodafone Cash     |
| 026, 056, 027, 057 | AirtelTigo Money |

This matters because MTN MoMo and Vodafone Cash have different API endpoints, authentication flows, and settlement timelines.

## What Would Change in Production

| Demo                         | Production                                      |
|------------------------------|-------------------------------------------------|
| In-memory Maps               | PostgreSQL (transactions) + Redis (idempotency) |
| `setTimeout` for retries     | SQS / BullMQ message queue                      |
| Simulated provider response  | Real MTN MoMo API (OAuth2 + REST)               |
| Plaintext seed passwords     | bcrypt hashes                                   |
| Single process               | Horizontally scaled behind a load balancer      |
| Console logging              | Structured JSON logs (Pino) → CloudWatch        |
